# Product

## Register

product

## Users

Common Good Interfaces is for curious builders: indie website makers, Cloudflare and Workers developers, agent-tool builders, and Manzanita-adjacent collaborators who want the web to feel executable again. They arrive with technical fluency and taste, but not necessarily full context on AgentAuth, Durable Objects, or the Common Good vocabulary.

## Product Purpose

The project documents and ships a small executable namespace for personal and public websites. A binlet is a URL-addressable capability with behavior, state, policy, and optional UI. V0 proves the primitive through `/common-good/counter`, a Cloudflare Worker starter that combines human HTML, JSON responses, discovery metadata, and Durable Object state.

Success means a builder can understand the idea quickly, believe it is technically real, and run or adapt the counter starter without the docs collapsing into either marketing fog or protocol ceremony.

## Brand Personality

Personality: curious, infrastructural, weird-good.

The docs should feel like practical developer documentation for a new web primitive, with enough personality to carry the neo-cgi-bin spark. They can be snazzy, specific, and a little strange, but they should stay useful at the point of implementation.

## Anti-references

Avoid anything normie: generic SaaS documentation, bland AI-platform gloss, sterile enterprise protocol pages, and default template aesthetics that erase the old-web insight. Also avoid pure retro-computing cosplay; the CGI-bin reference is an origin story, not a costume.

## Design Principles

1. Make the primitive feel executable: every major page should point back to URLs, routes, manifests, requests, state, and deployable code.
2. Personality serves comprehension: style can be memorable, but it should make the system easier to parse, not harder.
3. Show the governed edge: policy, receipts, moderation, consent, and ownership are core to the idea, not appendix material.
4. Keep V0 small and believable: the counter starter is the proof, and roadmap energy should not obscure what works today.
5. Invite curious builders in: explain unfamiliar terms without flattening the project into generic developer-platform language.

## Accessibility & Inclusion

Target WCAG AA for text contrast, focus states, keyboard navigation, and responsive layout. Respect reduced-motion preferences. Do not rely on color alone to convey status or category, especially in route, policy, and capability examples.
